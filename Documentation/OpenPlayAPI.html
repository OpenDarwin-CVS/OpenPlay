Introduction – What is OpenPlay?       Table of Contents

 

OpenPlay is a cross-platform networking API (Application Programming Interface) to allow Linux (coming soon), Macintosh, and Windows users to play a networked game together.  OpenPlay provides functions to do the following:

 

·      Configure network protocols.

·      Compile a list of computers hosting games.

·      Setup dialog boxes to host and join games.

·      Establish network connections.

·      Send data to other players in the game.

·      Receive data from other players in the game.

 

While OpenPlay does provide much networking help, it does not do everything a networked game needs.  Your game is responsible for the following:

 

·      Creating and displaying dialog boxes for hosting and joining networked games.

·      Handling events in dialog boxes.

·      Creating message data structures containing the sender, recipient, and the contents of messages between players.

·      Establishing the maximum number of players in a game.

·      Creating groups of players for team-based games.

 

Remember that OpenPlay is a low-level networking library designed to give a lot of control to game programmers.  It performs basic networking functions and nothing else, which makes OpenPlay a bit intimidating if you have never programmed a network game before.  If you are new to game networking, you may want to start with the NetSprocket API.  NetSprocket does more work for you, and it runs on both Macintosh and Windows computers.  The file NetSprocket1.7.pdf included in the OpenPlay SDK contains a reference to programming with NetSprocket.

 

OpenPlay uses net modules to implement the low-level networking.  Your game calls an OpenPlay function, and OpenPlay calls the appropriate function in the net module, which you can see in the diagram below:

The advantage of this approach is that your game does not need to know which net module the player has on his computer.  All your game has to do is call the OpenPlay functions, and the correct behavior will occur on the player’s machine.  Currently, OpenPlay has net modules for the TCP/IP protocol on Linux, Mac, and Windows computers along with a net module for the AppleTalk protocol on Macintosh computers.  If someone creates new net modules in the future, your OpenPlay code will still work on these new modules.

 

Conceivably a Mac could host a game on the Internet, which a player running Windows and another running Linux could join.  The Mac hosting the game could be connected to an AppleTalk local area network, and another computer on the network could join the game.  In this case, the four people playing the game would have four different OpenPlay net modules.  The host would use the Mac TCP/IP net module.  The Windows and Linux players would use the TCP/IP net modules for Windows and Linux respectively.  The second Mac would use the AppleTalk net module.  Your OpenPlay code will run on all four computers.

 

This manual covers programming with the OpenPlay API.  It does not cover writing an OpenPlay net module.  If you are interested in writing an OpenPlay net module, refer to the file Writing_OP_NetModules that comes with the OpenPlay SDK.

 

Before You Begin           Table of Contents

 

Before you write any code with OpenPlay, you must perform some preliminary work.  You must setup the OpenPlay libraries and net modules so your OpenPlay code will compile and run.  The sections below show you what you need to know for your code to compile and run properly.

 

Compiling the OpenPlay Libraries   Table of Contents

 

At this time there are two versions of the OpenPlay SDK, one version for people developing OpenPlay and one version for people who just want to use OpenPlay in their games.  The version for the people developing OpenPlay contains the source code while the other version contains the OpenPlay libraries and net modules.  If you are using the version of OpenPlay with the source code, you must manually create the OpenPlay libraries and net modules by compiling the OpenPlay projects that come with the OpenPlay SDK.  Right now, there are OpenPlay project files for CodeWarrior Pro 6 and CodeWarrior Pro 7 on Macintosh and Windows systems as well as Visual C++ for Windows.  This means you must have CodeWarrior Pro 6 or later or Visual C++ to help write OpenPlay; those of you who use an older version of CodeWarrior, Project Builder, or a non-Microsoft Windows compiler are out of luck. 

 

Compiling the OpenPlay libraries is easy to do in CodeWarrior, assuming you have a suitable version of CodeWarrior.  In the OpenPlay SDK, there exists a Project_Files directory.  Inside that directory are directories for CodeWarrior Pro 6 and 7; choose the one that matches your version of CodeWarrior.  Now you will see a list of project files.  To compile all the OpenPlay libraries and net modules, open the project file MasterBuild.mcp and build it.  Building the MasterBuild.mcp project will create all the OpenPlay libraries and net modules as well as creating executable files for the OpenPlay examples that come with the OpenPlay SDK.  The OpenPlay libraries and net modules will appear in the Targets subdirectory in the directory that contains the OpenPlay SDK.

 

If you want to compile the OpenPlay CodeWarrior projects separately, you can do so.  The list below contains a list of the projects:

 

·      NetSprocketTests.mcp, which contains a sample program for working with NetSprocket.

·      OpenPlayLib.mcp, which will create the OpenPlay library.  You must build this project before you build any of the OpenPlay sample projects.

·      OpenPlayMiniDemo.mcp, which contains a basic OpenPlay sample program.

·      OpenPlayTest.mcp, which contains another OpenPlay sample that’s a bit more sophisticated than the Mini Demo program.

·      OPEnumTest.mcp, which contains a sample of enumerating hosts in OpenPlay.

·      OPNetModulesMac.mcp, which will create the Mac OpenPlay net modules.

·      OPNetModulesWin.mcp, which will create the Windows OpenPlay net modules.

 

Like the CodeWarrior projects, the Visual C++ OpenPlay projects reside in the Project_Files subdirectory inside the directory containing the OpenPlay SDK.  There are two Visual C++ OpenPlay projects: OpenPlayLib.dsp, which will create the OpenPlay libraries, and OPNetModulesWin.dsp, which will create the Windows OpenPlay net modules.  Build these two projects in Visual C++, and you will be on your way to using OpenPlay in your Windows games.

Macintosh Users      Table of Contents

 

Macintosh users must perform the following steps to setup OpenPlay:

 

  1. Install the OpenPlay library.
  2. Install the OpenPlay net modules.
  3. Add the OpenPlay library to your project.
  4. Include the OpenPlay header file in your source code files.

 

Choosing which OpenPlay library to install can be confusing because there are four versions of the OpenPlay library on the Mac.  The list below contains the file names of the four versions:

 

·      OpenPlayLib, the release version of the OpenPlay library that runs on Mac OS 7, 8, and 9.

·      OpenPlayDebugLib, the debugging version of the OpenPlay library for Mac OS 7/8/9.

·      OpenPlayCarbonLib, the release version of the OpenPlay library that runs on Mac OS 8.6, 9, and X.

·      OpenPlayCarbonDebugLib, the debugging version of the OpenPlay library for Mac OS 8.6, 9, and X

 

As you can see from the list above, there are Carbon and Classic versions of the library, and each of those libraries has a debugging and a release version.  The Carbon versions work with Carbon programs that run on Mac OS 8.6, 9, and X.  The Classic versions run on Mac OS 7, 8 and 9 only.  You will most likely use one of the Carbon versions of the OpenPlay library because the Classic versions do not work with Mac OS X.  The debugging versions aid in debugging your OpenPlay games while the release versions run faster.

 

After deciding which OpenPlay library to use, place a copy of it in the folder where your game resides.  You should include only one version of the OpenPlay library in your game’s folder.  If you have multiple versions of the OpenPlay library in your game’s folder – for example, debug and release versions -- you can’t be sure which version the computer will use, and the results will be unpredictable.  You should use one of the debug versions while you’re writing your game.  If you’re writing a Carbon game to be compatible with Mac OS X, use the Carbon version of the debug library.  Otherwise, use the Classic version.  When you have your game’s networking running, test it with the release versions of the OpenPlay library.

 

The Macintosh version of OpenPlay includes TCP/IP and AppleTalk net modules.  The TCP/IP net module runs games on the Internet, and the AppleTalk net module runs games on an AppleTalk local area network.  The list below contains the file names of the OpenPlay modules:

 

·      AppleTalkModule, the release version of the OpenPlay net module for AppleTalk networks that runs on Mac OS 7, 8, and 9.

·      AppleTalkDebugModule, the debugging version of the AppleTalk net module for Mac OS 7/8/9.

·      AppleTalkCarbonModule, the release version of the OpenPlay net module for AppleTalk networks that runs on Mac OS 8.6, 9, and X.

·      AppleTalkCarbonDebugModule, the debugging version of the AppleTalk net module for Mac OS 8.6, 9, and X.

·      IPModule, the release version of the OpenPlay net module for TCP/IP networks that runs on Mac OS 7, 8, and 9.

·      IPDebugModule, the debugging version of the TCP/IP net module for Mac OS 7/8/9.

·      IPCarbonModule, the release version of the OpenPlay net module for TCP/IP networks that runs on Mac OS 8.6, 9, and X.

·      IPCarbonDebugModule, the debugging version of the TCP/IP net module for Mac OS 8.6, 9, and X.

 

When you compile the OpenPlay projects to create the OpenPlay libraries and net modules, the net modules appear in the Targets folder inside the folder containing the OpenPlay SDK.  If you are using the OpenPlay version with the prebuilt net modules, the modules appear in the mac_carbon and mac_classic folders.  You can find them in the folder OpenPlay Modules.  There will be four versions of this folder: Carbon versions for Mac OS X compatibility and Classic versions for older versions of Mac OS X.  Each of these versions has both a debugging and a release version.  Each version will contain the appropriate TCP/IP and AppleTalk net modules.  Select a version – during development you will probably go with the Carbon debugging version – of the OpenPlay Modules folder and move a copy of it to your game’s folder.

 

OpenPlay comes with a stub library to link into your game’s project in your compiler.  There are Carbon (OpenPlayCarbonStubLib) and Classic (OpenPlayStubLib) versions of the stub library.  Pick the appropriate one for your game and add it to the project in your compiler.  Doing this will link the OpenPlay library to your game so your game can use OpenPlay.

 

In your game’s source code files, include the file "OpenPlay.h" to access the OpenPlay functions in your programs.  Now you’re ready to begin writing OpenPlay networking code.

 

Windows Users                 Table of Contents

 

Windows users must perform the following steps to setup OpenPlay:

 

  1. Install the OpenPlay library.
  2. Install the OpenPlay net modules.
  3. Add the OpenPlay library to your project.
  4. Include the OpenPlay header file in your source code files.

 

The Windows version of the OpenPlay library has two versions.  The release version (OpenPlay.dll) contains the OpenPlay library.  The debugging version (OpenPlayDebug.dll) helps you debug your game’s networking during the development of your game.  Choose one version of the OpenPlay library and move a copy of it to the directory containing your game’s executable file.  If you have both versions in your game’s directory, you cannot be sure which version the computer will use.  During development you should use the debug version.

 

The Windows version of OpenPlay current has only one net module, a TCP/IP module for playing games on the Internet.  Just like with the OpenPlay library, the OpenPlay net module has two versions: a debug (TCPIPDebug.dll) and a release (TCPIP.dll) version.  If you manually created the net modules, they will appear in the directory with the name OpenPlay Modules in the Targets subdirectory in the directory containing the OpenPlay SDK.  If you use the OpenPlay version with the prebuilt net modules, the OpenPlay Modules folder appears in the Windows subdirectory.  Place a copy of one version of the OpenPlay Modules folder in the directory containing your game’s executable file.

 

In your game’s project, you must include the OpenPlay library so your OpenPlay code will compile.  When you create the OpenPlay libraries, the Windows version contains the file OpenPlay.lib, which contains a version of the OpenPlay library for linking.  Add the file OpenPlay.lib to your game’s project in your compiler.

 

In your game’s source code files, include the file "OpenPlay.h" to access the OpenPlay functions in your programs.  Now you’re ready to write some networking code with OpenPlay.

 

Using OpenPlay              Table of Contents

 

This section provides a guide to writing OpenPlay code in your game, and it contains several code examples to illustrate OpenPlay programming.  The code examples in this manual are for demonstration only.  They were written as a guide to write OpenPlay code, not as a working program for you to plug into your game.  You would need to write additional code to use the code snippets in your game.  The OpenPlay SDK contains sample programs ready to compile that you can use to experiment with OpenPlay.

 

OpenPlay Terminology    Table of Contents

 

Protocol – A protocol is a set of rules for the exchange of data between two computers.  Examples of protocols are TCP/IP for Internet networking and AppleTalk for networking Macintosh computers.

 

Module – In OpenPlay each protocol has its own module.  Modules let your OpenPlay code run across different network protocols.  For example someone could host a game on the Internet using the TCP/IP protocol and have people on the host’s local area network join the game using the AppleTalk protocol.

 

Endpoint – For a networked game, each computer playing the game is an endpoint.  You use endpoints to establish connections to the networked game and to transfer data between other machines.

 

Configuration – OpenPlay uses configurations to open endpoints, enumerate the hosts for the game, and configure the network protocols.

 

Enumeration – OpenPlay uses enumerations to retrieve a list of available hosts on the Internet or a local area network for your game.

 

Using OpenPlay in Your Game          Table of Contents

 

When the player wants to play a multiplayer networked game, he (or she) performs the following steps:

 

  1. He connects to the network.  In most cases, it involves going on the Internet, but a player can also connect to a local area network (LAN).
  2. He runs the game and chooses a multiplayer game.
  3. A list of online games appears.  The player can choose to play in one of these games, or he can choose to host a game.
  4. If the player chooses to host a game, he waits for other players to join the game.

 

Once there are enough players to play the game, the networking involves sending and receiving messages.  Every time the player performs an action in the game, such as moving, or firing a weapon, he sends a message to the other players detailing the action.  When the other players do something in the game, they send messages as well, which the player receives.

 

As the programmer, you must perform the following steps with OpenPlay to get the player into the online game:

 

  1. Create a configuration.
  2. Enumerate the hosts.
  3. Display a dialog box for the player to join or host a game.
  4. Create an endpoint.
  5. Create a connection to the game.

 

Creating Configurations  Table of Contents

 

The first thing you must do when programming with OpenPlay is create a configuration.  An OpenPlay configuration contains network protocol and game information.  In your game, you need a configuration to assemble a list of computers hosting games, to display dialog boxes to host and join games, and to open endpoints so a player can connect to the game. 

 

To create a configuration, call the function ProtocolCreateConfig( ).  The function takes seven parameters.  The first parameter is the network protocol type, which will either be ‘Inet’ for TCP/IP (Internet) or ‘Atlk’ for AppleTalk (LAN).  The second parameter is a four-character code that uniquely identifies your game.  The third parameter is the name of the game that will appear on the network.  The fourth parameter contains any data you want to supply for the configuration, and the fifth parameter is the length of that data.  The sixth parameter contains configuration options, which you only need to worry about if you want to override the default options.  The final parameter stores the configuration the ProtocolCreateConfig( ) creates.  The following function creates a configuration:

 

void CreateConfiguration(NMType protocolType, PConfigRef configuration)

{

      NMErr error;

 

      // The first nil means there’s no custom data to add to

      // the configuration.  The second nil means to use the

      // default configuration options.

error = ProtocolCreateConfig(protocolType, kGameID, kGameName,

                  nil, 0, nil, &configuration);

}

 

 

Enumerating Hosts           Table of Contents

 

At the start of the game, you must enumerate the hosts to determine which computers are currently hosting games.  The player can choose to join one of these games or choose to host a game.  Enumerating hosts requires the following steps:

 

  1. Call ProtocolStartEnumeration( ) to begin the enumeration.
  2. Call ProtocolIdleEnumeration( ) to give OpenPlay time to update the list of hosts.
  3. Call ProtocolBindEnumerationToConfig( ) to connect the enumeration to the configuration.  You call this function when the player chooses a game to join to connect the host to the player’s configuration.
  4. Call ProtocolEndEnumeration( ) to end the enumeration.

 

Like the function name indicates, you begin enumerating hosts by calling ProtocolStartEnumeration( ).  The function takes four parameters.  The first parameter contains the configuration you created by calling ProtocolCreateConfig( ).  The second parameter is an OpenPlay callback function.  I will show you how to write an enumeration callback function later in this section.  The third parameter holds data that will be passed to the callback routine. 

 

The fourth parameter specifies whether or not to begin an active enumeration.  If you pass the value TRUE, OpenPlay will actively search the network for hosts.  Otherwise, OpenPlay will use previously saved hosts, such as hosts saved in a preferences file.  Most of the time, you will want to use the value TRUE.  A game situation where you would use the value FALSE would be a player playing an online game, leaving that game, and looking for another game to join.  In this case, you could get away with just using the existing list of hosts, so you could pass the value FALSE.

 

OpenPlay needs processor time to update the list of hosts while you enumerate them.  The function ProtocolIdleEnumeration( ) gives OpenPlay the time it needs.  Call ProtocolIdleEnumeration( ) periodically while you enumerate hosts.

 

When the player chooses a game to join, you must bind the host of the game to the configuration you created.  Calling ProtocolBindEnumerationToConfig( ) does this for us.  The function is simple from a parameter standpoint.  Pass the configuration in the first parameter, and the host in the second parameter.

 

When the player has made his decision whether to host or join a game, call ProtocolEndEnumeration( ) to stop enumerating.  The following function performs an enumeration:

 

void EnumerateHosts(PconfigRef theConfiguration, void* callbackData,

Boolean searchNetwork)

{

      NMErr error;

      Boolean noSelectionYet = true;

      Boolean playerJoiningGame;   

 

ProtocolStartEnumeration(theConfiguration, EnumerationCallback,

                  callbackData, searchNetwork);

 

      DisplayJoinGameDialog( );

 

      while (noSelectionYet) {

            // Give OpenPlay time to look for hosts

error = ProtocolIdleEnumeration(theConfiguration);

            if (error != kNMNoErr)

                  return;

 

            ReadPlayerSelection( );

 

      }

 

      // Your dialog event handling code would

      // determine if the player did join the game

if (playerJoiningGame) {

            error = ProtocolBindEnumerationToConfig(theConfiguration,

                        hostID);

            if (error != kNMNoErr)

                  return;

      }

 

      error = ProtocolEndConfiguration(theConfiguration);

}

 

 

As you can see from the EnumerateHosts( ) function above, the ProtocolBeginEnumeration( ) function makes use of an enumeration callback function, which you need to write.  When OpenPlay finds a new host or finds that a host is no longer hosting a game, OpenPlay will call the enumeration callback function.  The enumeration callback function takes three parameters.  The first parameter contains data you supply to the callback routine.  Generally, games will pass their list of available hosts to the callback routine.  If your callback routine doesn’t need any data supplied to it, pass nil.  The second parameter is the enumeration command.  You can add, delete, or clear an enumeration.  The third parameter is the enumeration item.  It contains the host computer’s ID and name plus any optional data you want to add about the host computer.  For gaming purposes, you may want to include the number of players currently connected to the game and the maximum number of players the host can handle.

 

Your callback function has to be able to handle the three enumeration commands: add, delete, and clear.  Your game will maintain a list of hosts; how you choose to store the list is up to you.  Handling the add command adds a host to the list.  Handling the delete command removes a host from the list, and handling the clear command removes all the hosts from the list.  Here’s how an enumeration callback function looks:

 

void EnumerationCallback(void *inContext,

NMEnumerationCommand inCommand,

                        NMEnumerationItem *item)

{

      switch (inCommand) {

            case kNMEnumAdd:

                  AddHostToList( );

                  break;

 

            case kNMEnumDelete:

                  RemoveHostFromList( );

                  break;

 

            case kNMEnumClear:

                  ClearHostList( );

                  break;

 

            default:

                  break;

      }

 

}

 

OpenPlay Dialogs   Table of Contents

 

Your game has most of the responsibility when using dialog boxes for hosting and joining online games.  Most games create their own user interface to give the game a consistent look and feel.  In this case, you will be creating your own dialog boxes to host and join games.  You can create the dialog box using the operating system’s functions if you don’t want the hassle of making your own dialog boxes.  Then you call ProtocolSetupDialog( ) to fill the dialog box with its items like buttons, text boxes, and checkboxes.

 

Handling events and the player hitting items in the dialog box is your responsibility.  OpenPlay does have the functions ProtocolHandleEvent( ) and ProtocolHandleItemHit( ).  Unfortunately, the ProtocolHandleEvent( ) currently does nothing, and the ProtocolHandleItemHit( ) does nothing on TCP/IP networks, which is what most online games use.

 

When you’re finished with a dialog box you made using ProtocolSetupDialog( ), call ProtocolDialogTeardown( ) to dispose of the parts of the dialog that ProtocolSetupDialog( ) set up.  You are responsible for disposing of the dialog box.

 

Endpoints                  Table of Contents

After the player chooses to host or join a game, you must create an endpoint so he can connect to the game.  To create an endpoint, call the function ProtocolOpenEndpoint( ).  When calling ProtocolOpenEndpoint( ), you must tell OpenPlay to create either a passive or an active endpoint.  A passive endpoint listens for connections while an active endpoint initiates a connection.  If the player hosts a game, you should create a passive endpoint by passing the value kOpenNone as the final parameter to ProtocolOpenEndpoint( ). 

 

If you want to create an active endpoint, pass the value kOpenActive as the final parameter to ProtocolOpenEndpoint( ).  Normally, you don’t need to create active endpoints.  When a player joins an existing game, the host of the game creates an endpoint when it accepts the player’s request to join the game.  The function below creates an endpoint:

 

void CreateEndpoint(PconfigRef theConfiguration, void callbackData,

Boolean activeEndpoint)

{

      NMErr error;

      NMOpenFlags flags;

      PEndpointRef endpointToCreate;

 

      if (activeEndpoint)

            flags = kOpenActive;

      else

            flags = kOpenNone;

 

      error = ProtocolOpenEndpoint(theConfiguration, EndpointCallback,

                        callbackData, &endpointToCreate, flags);

 

}

 

In addition to calling ProtocolOpenEndpoint( ), you must write an endpoint callback function to create an endpoint.  This callback function will handle events such as connection requests, sending messages, and receiving messages.  The endpoint callback routine takes five parameters.  The first parameter contains the endpoint on which the routine works.  The second parameter contains data the callback function needs.  The third parameter specifies the network event for the callback routine to handle.  Currently there are eight events for your callback routine to handle: connection requests, datagram data, stream data, flow clear, accept complete, handoff complete, endpoint died, and close complete.  The fourth parameter contains an error code, and the final parameter contains additional data the callback may require.  The following function provides an example of the look of an endpoint callback function:

 

void EndpointCallback ( PEndpointRef inEndpoint, void* inContext,

                        NMCallbackCode inCode, NMErr inError,

void* inCookie ) 

{

      switch (inCode) {

            case kNMConnectRequest:

                  HandleConnectionRequest( );

                  break;

 

            case kNMDatagramData:

                  ReceiveDatagramMessage( );

                  break;

 

            case kNMStreamData:

                  ReceiveStreamMessage( );

                  break;

 

            case kNMFlowClear:

                  HandleFlowClear( );

                  break;

 

            case kNMAcceptComplete:

                  HandleAccept( );

                  break;

 

            case kNMHandoffComplete:

                  HandleHandoff( );

                  break;

 

            case kNMEndpointDied:

                  HandleDeadEndpoint( );

                  break;

 

            case kNMCloseComplete:

                  HandleClose( );

                  break;

 

            default:

                  break;

      }

 

}

 

When a player leaves the game, call ProtocolCloseEndpoint( ) to close the endpoint.

 

Connections             Table of Contents

If the player hosts an online game, his computer has the responsibility of accepting or rejecting the requests of other players to connect to the game.  The endpoint callback function will receive a connection request and must determine whether or not to accept the connection.  Reasons for rejecting a connection request include the game having the maximum number of connections, the incoming player’s connection speed being too low, or the player’s skill level being too low.

 

If the game accepts the incoming player’s connection request, call the function ProtocolAcceptConnection( ).  ProtocolAcceptConnection( ) creates an endpoint for the new player so you don’t have to explicitly create an endpoint for new players joining the game.  Because ProtocolAcceptConnection( ) creates an endpoint, it uses the same endpoint callback routine you wrote to create endpoints.  Call ProtocolRejectConnection( ) if the game rejects the connection request.

 

The function below handles a connection request:

 

void HandleConnectionRequest(PEndpointRef theEndpoint,

void* callbackData)

{

      NMErr error;

      Boolean requestAccepted;

 

      // Code to determine whether or not to accept the

      // connection omitted.

 

      if (requestAccepted) {

error = ProtocolAcceptConnection(theEndpoint,

EndpointCallback, callbackData, NULL);

      else

            error = ProtocolRejectConnection(theEndpoint,

callbackData);

 

}

           

Sending Messages           Table of Contents

In OpenPlay, there are two types of messages you can send: datagram and stream messages.  Datagram messages use the User Datagram Protocol (UDP).  Datagram data is unreliable, meaning the packet you send is not guaranteed to arrive at the destination.  However, datagram messages have less overhead, which means they generally reach the destination faster.

 

Stream messages use the Transmission Control Protocol (TCP).  TCP packets are reliable, meaning that the data is guaranteed to arrive at the destination.  The packets being guaranteed to arrive means that the sender will keep sending the packet until the recipient acknowledges that it received the packet.  TCP packets carry additional overhead to achieve higher reliability, which means they generally take more time to reach the destination.

 

You may be wondering which message type you should use in your game.  Turn-based games like board, card, and some strategy games should use stream messages.  In a multiplayer turn-based game, it is important that the players receive each move in the game.  Players in a turn-based game are already waiting for the other players to make their moves so they won’t notice the extra time it takes to send stream messages.

 

On the other hand, real-time games like action and real-time strategy games should use datagram messages.  Real-time games send messages multiple times per second so speed takes precedence over reliability.  If a message doesn’t reach its destination the first time around, it’s too late to resend the message. 

 

To send a datagram message with OpenPlay, call the function ProtocolSendPacket( ), which has four parameters.  Pass the endpoint sending the message in the first parameter.  The second parameter contains the message.  The third parameter contains the length of the message.  OpenPlay does not use the fourth parameter, but you must supply it.  Use 0 as the last parameter.  The following function sends a datagram message:

 

Boolean SendDatagramMessage(PEndpointRef sender,

GameMessagePtr message)

{

      NMErr error;

 

      error = ProtocolSendPacket(sender, message,

sizeof(GameMessage), 0);

 

      if (error == kNMNoError)

            return true;

      else

            return false;

}

 

GameMessagePtr is a pointer to a GameMessage data structure you create to store the information you want to send in a message.  If you want to use a different name than GameMessage, feel free to do so.  You know your game’s data needs better than anyone else, but some information you would need to send in a message include the sender, the recipient, an ID for the message, and game data describing the player’s move.

 

Sending a stream message in OpenPlay involves a call to the function ProtocolSend( ).  It has the same parameters that ProtocolSendPacket( ) does, which you can see in the code below:

 

Boolean SendStreamMessage(PEndpointRef sender, GameMessagePtr message)

{

      NMErr error;

 

      error = ProtocolSend(sender, message, sizeof(GameMessage), 0);

 

      if (error == kNMNoError)

            return true;

      else

            return false;

}

 

 

Receiving Messages                 Table of Contents

Sending messages is just one half of the game networking equation.  Receiving messages from other players is just as important.  If you can’t receive messages from other players, you might as well play a single-player game.

 

To receive datagram messages, call the function ProtocolReceivePacket( ).  It takes four parameters.  The first parameter is the endpoint of the recipient of the message.  The second parameter is the message to receive.  ProtocolReceivePacket( ) fills the parameter with the message’s data.  The third parameter is the size of the message.  You specify a message size on input, and ProtocolReceivePacket( ) returns the amount of data read in this parameter on output.  The final parameter are any flags that ProtocolReceivePacket( ) outputs. 

 

When there is no more data to read, ProtocolReceivePacket( ) returns the value kNMNoDataErr.  Call ProtocolReceivePacket( ) repeatedly until it returns a code other than kNMNoError.  The function below receives a datagram message:

 

void ReceiveDatagramMessage(PEndpointRef recipient)

{

      NMErr error = kNMNoError;

      GameMessage messageToReceive;

      NMFlags flags;

      unsigned long messageLength = sizeof(GameMessage);

 

      while(error == kNMNoError) {

error = ProtocolReceivePacket(recipient, &messageToReceive,

            &messageLength, &flags);

 

// Reset message length for next packet

messageLength = sizeof(GameMessage);

      }

 

}

 

Receiving stream data involves a call to ProtocolReceive( ), which has the same parameters that ProtocolReceivePacket( ) does.  Like ProtocolReceivePacket( ), ProtocolReceive( ) returns kNMNoDataErr when there is no more data to read.  Keep calling it until it returns an error code other than kNMNoError.  The function below receives a stream message:

 

void ReceiveStreamMessage(PEndpointRef recipient)

{

      NMErr error = kNMNoError;

      GameMessage messageToReceive;

      NMFlags flags;

      unsigned long messageLength = sizeof(GameMessage);

 

      while(error == kNMNoError) {

error = ProtocolReceive(recipient, &messageToReceive,

&messageLength, &flags);

 

// Reset message length for next packet

messageLength = sizeof(GameMessage);

      }

 

}

ProtocolSendPacket      Table of Contents

 

NMErr

ProtocolSendPacket( PEndpointRef endpoint,

                    void         *inData,

                    long         inLength,

                    NMFlags      inFlags

                  );

 

The ProtocolSendPacket( ) function sends a packet of data using the User Datagram Protocol (UDP).  Datagram data is unreliable, meaning the packet you send is not guaranteed to arrive at the destination.  However, sending datagram data is faster than sending stream data, making datagram data the better choice for realtime games. 

 

The size of the packet (the inLength parameter) must be less than the maximum packet size, which varies for each protocol.  The maximum packet size for the AppleTalk protocol is 576 bytes. The maximum packet size for the TCP/IP protocol depends on the low-level hardware; on an Ethernet network, the maximum packet size is 1500 bytes.

 

Parameter

Description

endpoint

The endpoint sending the packet.

inData

The packet to send.

inLength

The size of the packet.

inFlags

OpenPlay does not currently use the inFlags parameter.

ProtocolSend( PEndpointRef  endpoint,

              unsigned long inSize,

The ProtocolSend( ) function sends a packet of data using the Transmission Control Protocol (TCP).  ProtocolSend( ) returns the number of bytes that it actually sends.  TCP packets are reliable, meaning that the data is guaranteed to arrive at the destination.  This means that the sender will keep sending the packet until the recipient acknowledges that it received the packet.  The reliability of TCP packets makes them attractive for turn-based games, where it’s important that all the other players receive each player’s turn.

 

Parameter

Description

endpoint

The endpoint sending the data.

inData

The data to send.

inSize

On input, the inSize parameter contains the amount of data you want to send.  On output, ProtocolSend( ) returns the amount of data sent.

inFlags

Flags.
